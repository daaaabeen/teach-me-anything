# 模块 2：循环的解剖学

学习日期：2026-06-12 ｜ 测验：4/4 一次通过

## 核心模型

**一个完整的 loop = 触发器 + 可验证的目标 + 「规划→行动→观察→验证→终止」循环体。**

- **触发器**：消息 / 定时器 / 事件（CI 挂、新 issue）。cron 只覆盖这一块——cron + agent ≠ loop，还缺目标判定来闭合外循环
- **可验证目标**：机器可判真假（"测试全绿且 lint 零警告" ✓；"代码更优雅" ✗）。这是 loop 和瞎跑的分水岭，循环靠它知道何时停
- **验证与终止**：核心设计原则——**评判者和执行者分离**

## 评判者-执行者分离（本模块最重要）

写代码的 agent 不能自己宣布完成。原因：**犯错的模型和它的盲区共享同一套假设**——写代码时遗漏的边界条件，自查时同样遗漏（产生错误的思路 = 检查时用的思路）。独立模型判定 = 不同上下文 = 不相关的盲区。Claude Code `/goal` 即此实现：每轮由独立小模型检查完成条件。人类 code review 同理。

## 六大构件

| 构件 | 职责 | 形态 |
|------|------|------|
| 自动化任务 | 定时/事件触发 | Codex Automations、CC 定时任务 |
| 工作树 | 并行 agent 物理隔离，防互相覆盖 | git worktree |
| 技能 | 沉淀项目知识，每轮自动携带 | SKILL.md |
| 连接器 | 接外部系统 | MCP（Linear/Slack/DB） |
| 子代理 | 分工（写/查/验分离） | .claude/agents/ |
| 状态管理 | 跨轮跨 session 记忆 | markdown / 看板 |

抽掉状态管理 → 跨天任务无法保证方向一致（今天不知道昨天决定了什么，可能推翻重来原地打转）。

## 典型组合（Osmani 晨间流程）

定时触发 → triage 技能读 CI 失败/新 issue → 每个可修项开独立 worktree → 子代理起草修复 + 另一子代理验证 → 连接器开 PR/更新工单 → 搞不定的进人工队列。**你睡觉时循环在跑。**

## 元认知锚点

本学习工作流就是六大构件的实例：/learn 是技能、测验 gate 是评判者-执行者分离、state.md 是状态管理、断点续学是状态落盘的效果。
