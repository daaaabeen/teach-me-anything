# 模块 3：记忆与状态——循环的持久化

学习日期：2026-06-12 ｜ 测验：4/4 一次通过

## 核心模型

agent 的两种记忆 = **内存 vs 硬盘**：

| | 上下文窗口 | 磁盘（文件/仓库） |
|--|-----------|------------------|
| 寿命 | 易失（session 结束蒸发） | 持久 |
| 容量 | 有限 | 无限 |
| 成本 | 贵（每轮全量重读） | 便宜（按需读） |
| 质量 | 超长时注意力退化（context rot） | 不退化 |

**「Agent 会忘，但仓库不会。」**（Osmani）

## 为什么必须落盘（四个递进理由）

1. **蒸发**：每轮冷启动，不落盘就从零开始
2. **装不下**：跨天任务历史超出窗口容量
3. **会变质**：装得下也记不清——超长上下文注意力衰减
4. **方向漂移**：不知道昨天的决策，今天可能翻案重来、原地打转

大窗口解决不了 1（易失性是本质），所以「窗口够大就不用落盘」是错的。

## 每轮的标准模式

> **读状态 → 做一个增量 → 写状态 → 退出。**

- 状态文件是**轮与轮之间的契约**
- **状态先行**（增量完成立即写，不攒到最后）：循环随时可能中断，攻到哪写到哪才能任意断点恢复
- 状态要**人可读**（markdown）：循环出问题时由人审计修复

## 状态文件该记什么

- 目标与计划、已完成清单、**决策及理由**（防翻案）、**下一步指针**
- **试过但失败的路**——失败是付过钱的信息。不落盘 → 每轮冷启动重新「发现」同一个好主意 → 反复撞同一堵墙（诊断特征：连续 N 轮重复同一失败方案）

## 记忆系统的分工

- **对话历史**：被动残留，有损（自动压缩）、不可移植 → 不配当状态存储
- **state 文件**：主动提炼的当前快照 + 未来指针（朝前看）
- **git commit log**：不可变的过去，审计轨迹（朝后看）

## 三层范式定位

prompt engineering（一句话）⊂ context engineering（一个 session 的窗口）⊂ **loop engineering（跨 session 活下来什么）**——状态落盘是最外层的地基。
