# 速查：AI Agent 循环工程（Loop Engineering）

> 结题日期：2026-06-12 ｜ 层级：L2 能讲 ｜ 五模块全过（测验 20/20），费曼终验通过

## 一句话

**人不再提示 agent，而是设计「提示 agent 的循环」**——一个 loop = 触发器 + 可验证目标 + 「规划→行动→观察→验证→终止」，agent 在其中无人值守地工作，人退到最外层做循环设计者和最终验证者。

## 核心模型

**两个循环**：内循环 = 一条消息内的「思考→工具→观察」（agentic loop，机制，2022 ReAct 起）；外循环 = 「下一条消息谁来发」（loop engineering 的主战场，2026.6 得名）。

**判断是不是 loop 的刀**：下一步是代码写死的（workflow/cron），还是模型看完结果现场决定的（loop）。

**六大构件**：自动化任务（触发）、工作树（并行隔离）、技能（项目知识）、连接器（MCP）、子代理（分工）、状态管理（跨轮记忆）。

**评判者-执行者分离**：写代码的不能自评——犯错的模型和它的盲区共享同一套假设；独立验证者 = 不相关盲区。可推广到组织层面（跨团队 loop 协作）。

**状态落盘**：「agent 会忘，但仓库不会」。每轮 = 读状态→做增量→写状态→退出；状态先行保证任意断点可恢复。必记：目标、决策及理由、**失败尝试**（付过钱的信息）、下一步指针。

## 失败模式 × 防护（速查表）

| 失败 | 诊断特征 | 主防护 |
|------|----------|--------|
| 死循环 | 到不了终止条件 | 可验证目标 + 迭代上限 |
| 空转（最隐蔽） | 看着忙、进度不动（修A挂B摆动） | 无进展检测（唯一能提前止损的） |
| 目标漂移 | 修bug修成大重构 | 目标入状态、每轮重新注入 |
| 上下文溢出 | 压缩丢约束后失控 | 状态落盘 + 增量退出 |
| 成本失控 | 单agent 4×、多agent 15× | token 预算熔断 |

## 关键结论

- **瓶颈转移**解释一切时机：模型弱拼 prompt → 窗口大拼 context → 模型可靠拼 loop。每波 XX engineering 都是当时瓶颈的路标
- 什么活儿交给 loop：**重复发生 + 行为标准明确 + 目标可验证**（如值班修 bug）；临时几轮对话能解决的自己来
- 人不可让渡的责任：**最终验证 + 理解系统**。理解债务复利（没读懂的代码上长新代码），验收是还债机制
- 「带着判断力设计循环是解药，为逃避思考设计循环是加速器」——Osmani

## 易错点（亲测踩过）

1. ❌「死循环 = 有进展但停不下来」→ 死循环的定义只关乎终止条件；**空转才是更危险的**（看起来在干活、发现晚烧得多）
2. ❌「标准化任务不产生理解债务」→ 债务来自「合并没读过的产出」，与任务类型无关；适合 loop 的真因是目标可验证 + 范围有界
